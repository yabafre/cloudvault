## Dev Notes

### Architecture (LAW)

- **Data Layer ¬ß2.2** ‚Äî One `.prisma` file per model in `apps/api/prisma/schema/`, kebab-case names (`refresh-token.prisma`). Layout matches the architecture diagram verbatim.
- **Prisma version** ‚Äî 7.7.0 per architecture table. In v7 the folder layout is GA, so no `previewFeatures = ["prismaSchemaFolder"]` is required.
- **No schema behavior change** ‚Äî This story is a pure refactor + tooling upgrade. No column added, renamed, or dropped. No index altered. `@@map` and `@map` directives preserved so the database schema stays byte-identical.
- **Backend layer boundaries ¬ß3** ‚Äî `PrismaService` stays the only place instantiating `PrismaClient`. Consumers keep importing it via Nest DI, never calling the client directly.

### Prisma 6 ‚Üí 7 breaking changes that apply here

1. **Generator renamed**: `prisma-client-js` is deprecated; `prisma-client` (Rust-free, TS-only) is the v7 default.
2. **`output` is mandatory**: no more default emission to `node_modules/.prisma/client`. We target `apps/api/src/prisma/generated/` so the emitted code ships with the NestJS build and is reachable via the `@/prisma/*` path alias.
3. **Driver adapter is required**: `new PrismaClient()` without an adapter throws at runtime in v7. We use `@prisma/adapter-pg` since we run against Postgres (Neon). No change to `DATABASE_URL` ‚Äî the adapter accepts the same pooler connection string (`channel_binding=require&sslmode=require`) as-is.
4. **Import paths change**: `@prisma/client` ‚Üí `@/prisma/generated/client`. The existing TS path alias `@/*` ‚Üí `src/*` (see `apps/api/tsconfig.json`) makes this clean across the codebase.
5. **`Prisma.validator` removed**: not used anywhere in the repo (verified by grep). Nothing to migrate.

### Schema folder content mapping (zero behavior delta)

Source file `apps/api/prisma/schema.prisma` (current monolith) splits as follows ‚Äî each block copied verbatim, comments preserved:

- `schema.prisma` (new) ‚Üê lines 1‚Äì11 (datasource + generator), plus `output` and `provider = "prisma-client"`
- `user.prisma` ‚Üê `AuthProvider` enum (lines 13‚Äì17) + `User` model (lines 29‚Äì49)
- `file.prisma` ‚Üê `FileStatus` enum (lines 19‚Äì27) + `File` model (lines 67‚Äì90)
- `refresh-token.prisma` ‚Üê `RefreshToken` model (lines 51‚Äì65)

Cross-file relations (`User.files`, `User.refreshTokens`, `File.user`, `RefreshToken.user`) stay valid ‚Äî Prisma merges every `.prisma` file in the schema folder into one logical schema at generate time.

### `PrismaService` refactor ‚Äî target shape

```ts
import { Injectable, Logger, OnModuleDestroy, OnModuleInit } from '@nestjs/common';
import { PrismaPg } from '@prisma/adapter-pg';
import { PrismaClient } from '@/prisma/generated/client';

@Injectable()
export class PrismaService extends PrismaClient implements OnModuleInit, OnModuleDestroy {
  private readonly logger = new Logger(PrismaService.name);

  constructor() {
    const adapter = new PrismaPg({ connectionString: process.env.DATABASE_URL as string });
    super({
      adapter,
      log: process.env.NODE_ENV === 'development'
        ? ['query', 'info', 'warn', 'error']
        : ['error'],
    });
  }

  async onModuleInit() { /* unchanged */ }
  async onModuleDestroy() { /* unchanged */ }
  async cleanDatabase() { /* unchanged ‚Äî still deletes file then user */ }
}
```

### Debug Log

- `pnpm install` initially did not materialize `prisma@7.7.0` in `node_modules/.pnpm` even though the lockfile was updated ‚Äî only Prisma 6.19.1 remained in the store. Resolved by a clean `rm -rf node_modules && pnpm install`. Root cause suspected: shared `pnpm` store state across the main project and worktree checkouts, stale before the frozen-lockfile toggle.
- First `pnpm db:generate` failed with "Could not find Prisma Schema". Prisma 7 does **not** auto-discover the schema folder layout the way v6 preview did; the tool requires explicit pointing via `prisma.config.ts`. Added that file with `schema: prisma/schema`.
- Second attempt failed with `P1012: datasource property 'url' is no longer supported in schema files`. Prisma 7 moves the datasource URL out of `schema.prisma` into `prisma.config.ts` (`datasource.url = env('DATABASE_URL')`). Removed the `url` line from `schema.prisma/datasource` block.
- Generated client used `import.meta.url` (ESM-only), which NestJS's CommonJS TypeScript config rejected. Fix: added `moduleFormat = "cjs"` to the generator block. Client was re-emitted as CJS-compatible.
- Jest failed to resolve the `@/` path alias (tsconfig paths are not consumed by jest automatically) and then failed on the generated code's `.js` extension imports. Added `moduleNameMapper` entries for the tsconfig aliases and a `^(\\.{1,2}/.*)\\.js$` ‚Üí `$1` rule to translate ESM-style `.js` imports to their `.ts` source at resolution time.
- `prisma migrate status` required the `DATABASE_URL` env to be loaded; using `pnpm dotenv -- ...` (matching the root script pattern) solved it.
- Final Neon smoke ran from the compiled `dist/` because `ts-node` with the generated `.js` import pattern trips over the same NodeNext resolution issue that jest had ‚Äî the compiled build handles it correctly via NestJS's webpack-less `nest build`.
